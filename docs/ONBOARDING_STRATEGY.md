# クライアントオンボーディング戦略

> 最終更新: 2026-03-02

## 概要

Exposaasを新規クライアント（中古車輸出業者）に展開する際のオンボーディング戦略。
対象規模: 初期5〜10社。

## 1. MOVFAX導入 + メール設定

### 背景

- オークション会社からの請求書は **メール直送**（USS等）と **FAX** の2パターン
- 物理FAXを使っている会社が多い → メール自動取込パイプラインに乗らない
- **MOVFAX**（クラウドFAXサービス）を導入すれば、FAX受信がPDFメール転送になる

### 決定事項

- **MOVFAXの導入はオンボーディングの必須ステップとする**
- 費用（月額980円〜）は **クライアント負担**
- FAX番号はそのまま使える（ポータビリティ）ので、クライアントの業務フロー変更はほぼゼロ
- MOVFAXの受信PDF送信先に、Exposaas側のメールアドレスを追加登録する

### クライアントへの説明ポイント

- 「FAX番号は変わりません」
- 「届いたFAXが自動でPDFになり、システムに直接取り込まれます」
- 「手動でPDFをアップロードする手間がなくなります」
- 「月額980円で業務効率が大幅に改善されます」

## 2. メール自動取込アーキテクチャ

### 決定事項: 独自ドメイン catch-all 方式

各クライアントのGmailにOAuth接続するのではなく、**こちらが管理する独自ドメインのメールアドレスを送信元（MOVFAX/USS等）に直接登録する。**

```
USS / MOVFAX / 各オークション
        │
        ▼ メール送信先として登録
  clientA@inv.exposaas.com ──┐
  clientB@inv.exposaas.com ──┼─→ Cloudflare catch-all ──→ 1つのGmail ──→ Exposaas
  clientC@inv.exposaas.com ──┘
                                                          OAuth接続は1回だけ
```

### なぜこの方式か

| 比較項目 | 各社のGmailにOAuth | 独自ドメイン catch-all |
|----------|-------------------|---------------------|
| OAuth接続 | 会社ごとに必要 | **1回だけ** |
| クライアントの手間 | OAuth同意画面の理解が必要 | **なし（こちらが設定）** |
| Google OAuth審査 | 必要（多ユーザー時） | **不要** |
| token失効対応 | 会社ごとに発生 | **1アカウント分のみ** |
| コスト | 無料 | **無料（Cloudflare Email Routing）** |

### 初期セットアップ（1回のみ）

1. Cloudflare で `inv.exposaas.com`（または類似のサブドメイン）のcatch-allルールを設定
2. 転送先として専用Gmail（例: `exposaas.invoices@gmail.com`）を作成
3. そのGmailに対してOAuth接続を1度だけ実行
4. emailIngestion worker を修正: `To` アドレスから companyId を解決するロジックを追加

### 新規クライアントのオンボーディング手順（毎回）

1. Exposaasにクライアント会社を作成 → companySlug を決定（例: `warldauto`）
2. 以下のメール送信先をオークション側に登録:
   - MOVFAX管理画面 → 受信PDF送信先 → `warldauto@inv.exposaas.com`
   - USS会員ページ → メール通知設定 → `warldauto@inv.exposaas.com`
   - その他オークション → 同様
3. **クライアントも受信したい場合**: MOVFAX/USS側で送信先を2つ設定（クライアントのメール + Exposaasのメール）

**これだけで、そのクライアントの請求書が自動的にExposaasに流れ始める。**

### 会社識別ロジック

```
受信メールの To: warldauto@inv.exposaas.com
                  ^^^^^^^^
                  この部分で GmailAccount → companyId を解決
```

DB上は `GmailAccount.emailAlias` のようなフィールドで `warldauto` → `companyId: 5` のマッピングを保持する。

## 3. 既存Excel在庫台帳の移行（要設計）

### 課題

- 各社が独自のExcelシート（在庫台帳）で車両管理をしている
- **列名・列順・管理項目が全社異なる**（統一フォーマットは存在しない）
- Exposaasを使い始めるには、既存データを移行する必要がある
- 移行なしだと「過去の車両データがないシステム」になり、導入障壁が高い

### 現状の仕組み

- CSVアップロード機能は実装済み（`addDocument.js` → `vehicle.mjs` worker）
- ただし **固定のカラムマッピング** を前提としている（`chargeMapping.mjs` の `METADATA_CSV_MAP`）
- 各社バラバラのExcelをそのままアップロードしても正しくマッピングされない

### 考えられるアプローチ

#### A. AI列マッピング（推奨検討）

```
クライアントのExcel → Geminiに列名を見せる → Exposaasの列にマッピング提案
                                               → ユーザーが確認・修正
                                               → マッピングルールを保存
                                               → 以降は同じルールで自動変換
```

- Geminiに「このExcelの列名をExposaasのスキーマにマッピングして」と聞く
- ユーザーが確認（HITL）してマッピングを確定
- 確定したマッピングは会社ごとに保存 → 2回目以降は自動

#### B. 手動マッピングUI

- アップロード後に「この列はどのフィールドに対応しますか？」のUI
- ドロップダウンで1列ずつ対応付け
- 確定後にインポート実行

#### C. BPO側で手動変換

- オンボーディング時にBPO担当者がExcelを確認
- Exposaasのテンプレートに手動で合わせる
- スケールしないが、初期5社程度なら現実的

### 推奨: フェーズドアプローチ

1. **初期（〜5社）**: C（手動変換） — 各社のExcelを理解し、パターンを把握
2. **中期（5〜10社）**: B（手動マッピングUI） — パターンが見えてきたらUI化
3. **将来（10社〜）**: A（AIマッピング） — 十分なパターンが蓄積されたらAI化

## オンボーディングチェックリスト（新規クライアント）

```
□ Exposaasに会社アカウント作成（companySlug決定）
□ 管理者ユーザー作成、ログイン確認

□ MOVFAX導入（クライアント負担、月額980円〜）
  □ MOVFAX契約、既存FAX番号のポータビリティ設定
  □ 受信PDF送信先に {companySlug}@inv.exposaas.com を追加
  □ テストFAX送信 → Exposaasに届くことを確認

□ メール直送設定（USS等）
  □ USS会員ページでメール通知先に {companySlug}@inv.exposaas.com を追加
  □ その他オークションも同様

□ 既存在庫台帳の移行
  □ クライアントからExcelファイルを受領
  □ 列マッピング確認・変換
  □ CSVインポート実行
  □ データ確認（車両数、主要フィールドの整合性）

□ 動作確認
  □ テスト請求書がメール経由で自動取込されることを確認
  □ 分類 → 抽出 → レビュー → 車両作成の一連フローを確認
  □ クライアントにレビュー操作をレクチャー
```
