# Product Strategy Case Studies

> **Read this when:** making product/feature decisions, prioritizing what to build next, deciding scope for external launches.

## Core Pattern (Both Companies)

1. **1つの機能を完璧にする** — 電話応対だけ（Toma）、文字起こしだけ（Abridge）
2. **口コミで広がる** — 営業ゼロ。最初の顧客が次の顧客を連れてくる
3. **顧客が「次これもできない？」と言ってから拡張** — 機能追加は既存顧客のアップセル手段であり、新規獲得手段ではなかった

**Exposaasへの翻訳:** 「落札書 → 在庫台帳」の自動化、これ1つだけで外部1社に使ってもらう。インボイス生成もバイヤー通知もブッキングリマインダーも今は作らない。

## Toma: 18ヶ月で$17M調達

**何をやっているか:** 米国の自動車ディーラー向けAI音声エージェント。電話に出てサービス予約を取る。

### 創業者
- Monik Pamecha & Anthony Krivonos。AIエンジニア（Scale AI, Uber, Lyft, Amazon出身）
- 技術力はあったがドメイン知識ゼロ。**自動車業界の人間ではなかった**
- Exposaasとの違い: Tomaは技術者が業界を探していた。Exposaasは業界人が技術を作っている。スタート地点としてExposaasの方が強い

### フェーズ1: 技術だけあってドメインがない（2024年初頭）
- 汎用の音声AIを構築し、金融・医療などいろいろな業界で試した
- 何もしっくりこなかった。規制が多い金融・医療はAI導入が遅い

### フェーズ2: 偶然の発見と検証（2024年夏前）
- ディーラーから電話がかかってきた：「電話が殺到して溺れてる」
- 全米のディーラーに電話をかけ続け、**45%しか応答されない**ことをデータで証明
- 主観的な「困ってる」ではなく、数字で問題の大きさを示した

### フェーズ3: 現場への没入（2024年夏）
- オクラホマとミシシッピのディーラーに乗り込み、タイヤ交換の手伝いや電話応対を実施
- a16zの投資担当は「彼らは事実上ディーラーに住み込んでいた」と表現

### フェーズ4: One-Size-Fits-One
- 「汎用音声AI」ではなく、**各ディーラーごとに別のAIエージェントを作る仕組み**
- 各ディーラーの実際の顧客コール1-2週間分でトレーニング
- ディーラー固有のナレッジベース（サービス価格、営業時間、挨拶の仕方）をAIに付与
- 成功するデプロイは**新入社員をトレーニングするプロセスを模倣**する

### フェーズ5: 口コミ爆発
- 最初の1軒でうまくいったら、そのディーラーが別のディーラーを紹介
- 営業ゼロ。最初のセールス担当者を雇ったのはSeries A（$17M）の数週間前
- 業界インフルエンサー（Car Dealership Guy / Yossi Levi）が投資家にもなり認知加速
- **100軒以上に広がった時点でも、機能はほぼ「電話応対+予約」のみだった**
- リコール管理、パーツ問合せ、ノーショー防止、Inboxなどは全部100軒に広がった後に順次追加

### 技術スタック
- TypeScript全スタック。Next.js, Bun, T3スタック（React, Prisma, PostgreSQL, NextAuth, tRPC）
- 旧世代IVRは決定論的な分岐ツリー。Tomaは確率的でコンテキスト認識型
- 例: 顧客が「エンジン警告灯が点いてるのとレンタカーが欲しい」→ AIは2つの要件を同時解析してDMSのスケジューラーから両条件に合う空きを検索

## Abridge: 7年で$5.3B

**何をやっているか:** 医療面談の音声をAIで文字起こし → SOAPノート自動生成 → 電子カルテ(Epic)に直接反映。

### 創業者
- Shiv Rao（CEO）: 現役循環器科医。臨床医としてドメイン知識
- Sandeep Konam: CMUロボティクス修士。ML/プロダクト実装力
- Florian Metze: CMU言語技術研究所研究員。音声認識専門
- **3つの専門性の交差点:** 臨床医（ドメイン）x 音声認識研究者（技術）x ML/プロダクト（実装）

### フェーズ1: 個人の痛みから出発（2018年）
- Raoは毎晩帰宅後にその日の全患者カルテを書き起こす「パジャマタイム」問題に執着
- IVF治療中の妻が面談後に話された内容をほとんど覚えていなかった経験も着想に
- 最初のプロダクトは予約を録音して文字起こしと定義を提供するシンプルなアプリ

### フェーズ2: 消費者アプリでデータ蓄積（2018-2022年）
- **最も巧みな戦略:** 無料の文字起こしアプリを配布してデータを蓄積
- 4年間「会話を録音して文字起こしする」だけ。機能は増やさなかった
- この期間に**150万件の医療会話データ**を蓄積 → 後に最大のモートに
- 直接収益は少なくても、データセット構築が本当の目的だった

### フェーズ3: 独自技術パイプラインの構築
- 汎用LLMに依存せず自社フルスタック構築
- 独自ASR: 医療用語・薬品名・略語に特化。28言語対応
- 独自ノート生成LLM: 医療データで訓練した自社モデル
- **Linked Evidence:** ノートの任意の箇所をハイライトすると、元の音声トランスクリプトの根拠を表示。これが信頼獲得の決定打
- GoogleやOpenAI Whisperを上回る精度（単語エラー率16%削減、医療用語エラー45%削減）

### フェーズ4: Epic統合（2024年〜、爆発の引き金）
- EpicはUS病院の42%が使う最大手電子カルテ。ここに直接統合
- 録音が終わるとノートがEpic内で医師を待っている状態に
- 「使わない理由がない」状態を作った

### フェーズ5: ドキュメンテーション → 収益サイクルへピボット
- 「カルテを書く」→「カルテを書きながら正確な請求コードも自動生成して病院の収益を直接改善」に拡張
- 導入の意思決定者が医師個人 → 病院経営陣に変わり、取引規模が桁違いに拡大
- ARR: 2024年末$60M → 2025年Q1に$117M。評価額4ヶ月で$2.75B → $5.3B

## Exposaasへの教訓

### Tomaから
| 教訓 | Exposaasでの意味 |
|------|------------------|
| ドメインがなければ刺さらない | Exposaasはドメインを既に持っている。Tomaより有利 |
| 最初のセールスは顧客が来てくれた | 業界内の信頼と口コミが全て。営業力ではない |
| One-Size-Fits-One | 会社ごとのカスタムフィールド、名寄せルールはまさにこれ |
| 創業→Series A: 18ヶ月、チーム2→20人 | 小さく始めて速く回す |

### Abridgeから
| 教訓 | Exposaasでの意味 |
|------|------------------|
| 自分の痛みを解決するところから始めた | Exposaasも全く同じ |
| 最初の無料プロダクトはデータ収集装置 | 最初の外部顧客から得るゴールデンデータも同じ役割 |
| Linked Evidence（根拠の紐付け）が信頼の鍵 | 「AIが抽出した値 → 元の落札書のどこに書いてあるか」を紐付けで見せれば信頼が劇的に上がる |
| 既存システム（Epic）への統合が採用を爆発させた | 顧客の既存の在庫管理ツール/Excelに寄り添うところから始める |
| ドキュメンテーション→収益への直接貢献でピボット | 「在庫入力自動化 → インボイス自動生成（直接売上に貢献）」がこれに相当 |

### なぜ「1つで広げる」が正しいのか

1. **信頼は機能数ではなく、1つの機能の完璧さで生まれる** — Tomaの最初のディーラーは「電話応対がうまくいった」から紹介した。「10機能ある」からではない
2. **機能が多いと導入ハードルが上がる** — 「請求書の自動読み取りだけ試してみて」は簡単。「業務全体をこのシステムに移行して」は怖い
3. **早く広げることでデータが集まり、データが次の機能を可能にする** — Abridgeは文字起こしだけの期間に150万件のデータを貯めた。そのデータがLinked Evidenceやレベニューサイクル管理を実現した

### 機能拡張のタイミング

機能を追加するのは、外部顧客が「これの次に、〇〇もできない？」と聞いてきた時。
Tomaもディーラーの方から声がかかることで次の機能が決まった。顧客が次に何を欲しがるかは、顧客に聞くまでわからない。
